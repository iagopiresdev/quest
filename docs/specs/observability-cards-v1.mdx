# Observability Cards v1

## Context

Observability sinks existed before this spec: webhook, Telegram, Slack, Linear, OpenClaw. All sinks shared a single plain-text formatter (`formatSinkTextMessage`) that returned a flat key/value block. Functional for diagnostics, flat for daily use. The daemon now emits eight event types per party lifecycle; an operator who subscribes to all of them gets a stream of look-alike messages.

Telegram is the only sink on the list with native support for rich presentation in the format we already control (HTML parse mode). It is also the sink most operators actually read on their phone, so it is the first place a presentation pass pays for itself.

## Problem

- Plain-text messages bury the event type at line 2 instead of the top.
- No visual distinction between dispatch, success, and failure.
- User-supplied strings (party names, error messages) are interpolated verbatim, so a hostile repo name would land in the sink unescaped. Not exploited today but a real injection surface once Markdown/HTML formatters arrive.
- Setup had no single documented command-first path for richer Telegram formatting. Operators had to know to run `observability telegram upsert --parse-mode HTML` manually.
- When a local OpenClaw config already contains a Telegram bot token, the operator had to re-enter it for Quest Runner.

## Goal

Ship a Telegram-only HTML card builder with RPG flavor copy, route the Telegram sink through it when `parseMode === "HTML"`, and document a command-first path that can reuse the OpenClaw bot token and default chat id.

## Non-Goals

- No Slack Block Kit cards in v1. The plain-text formatter remains the portable fallback and the default for Slack + webhook.
- No per-event opt-out (e.g., "show cards for failures but plain text for success"). All-or-nothing per sink.
- No Markdown/MarkdownV2 card path. HTML is the only rich mode supported today for Telegram.
- No translation or localization. Copy is English-only.
- No OpenClaw import for Slack / Linear. The pattern generalizes but only Telegram is wired in v1.

## Addendum: Linear RPG Markdown Cards (2026-04-16)

Linear issue comments render GitHub-flavored Markdown, so the card builder pattern extends cleanly. The Linear sink now accepts `useRpgCards: true` (CLI flag `--rpg-cards` on `observability linear upsert`), and when enabled routes events through `formatLinearCard` instead of the plain-text formatter.

The RPG copy (`⚔️ Quest Accepted`, `💀 Party Wiped`, etc.) matches the Telegram card builder 1:1 so the two sinks feel like the same product. Linear cards use `##` headings, italic flavor taglines, bulleted fact lists, fenced code blocks for errors, and a local clock footer.

Back-compat: existing Linear sinks with `useRpgCards` unset keep the plain-text formatter. No schema migration.

## Domain Impact

- `src/core/observability/` gets one new bounded helper (`sinks/telegram-card-builder.ts`). Existing schemas unchanged except for making `ObservableDaemonEvent.specFile` nullable, which was already broken for `daemon_budget_exhausted` and is a correctness fix.
- `detectSinkSetup` can read sink-related defaults for agent-driven setup and doctor checks.
- The setup flow now reads `$HOME/.openclaw/openclaw.json` as a side effect of running `quest setup`. Failure modes (missing file, malformed JSON) are handled silently; detection returns `null` fields.

## User Or Operator Workflow

Command-first path:

```
jq -r '.channels.telegram.botToken' "$HOME/.openclaw/openclaw.json" \
  | quest secrets set --name quest-telegram-bot-token --stdin

CHAT_ID=$(jq -r '.channels.telegram.allowFrom[0]' "$HOME/.openclaw/openclaw.json")

quest observability telegram upsert \
  --id daemon-telegram \
  --chat-id "$CHAT_ID" \
  --bot-token-secret-ref quest-telegram-bot-token \
  --parse-mode HTML \
  --events daemon_dispatched,daemon_landed,daemon_failed,daemon_budget_exhausted,daemon_recovered,daemon_party_created,daemon_party_resting,daemon_party_resumed
```

This sequence:

1. Writes the detected OpenClaw bot token into the secret store under `quest-telegram-bot-token`.
2. Upserts a Telegram sink with `parseMode: "HTML"`, `botTokenSecretRef: "quest-telegram-bot-token"`, and the imported chat id.

Operators who already ran `quest setup` can opt in manually:

```sh
quest secrets set --name quest-telegram-bot-token --stdin < <(printf '<bot-token>')
quest observability telegram upsert \
  --id daemon-telegram \
  --chat-id <chat-id> \
  --bot-token-secret-ref quest-telegram-bot-token \
  --parse-mode HTML \
  --events daemon_dispatched,daemon_landed,daemon_failed,daemon_budget_exhausted,daemon_recovered,daemon_party_created,daemon_party_resting,daemon_party_resumed
```

## Data And State Changes

- `observableDaemonEventSchema.specFile` changed from required `nonEmptyString(240)` to nullable. Existing delivery records keep the old string values. New records from `daemon_budget_exhausted` and the three party-admin events persist `null`.
- Sink config format unchanged (`parseMode` was already a supported field on `TelegramSink`).
- No migration required.

## Command Surface

No new top-level commands. The existing `observability telegram upsert --parse-mode HTML` already worked before this spec — the spec ships the presentation layer that makes it worthwhile.

## Observability

The HTML card path is the observability layer itself. We add:

- `formatTelegramCard` renders an `<i>` flavor line per daemon event type. The test suite asserts every shipped daemon event has a distinct header and flavor so future additions cannot forget to add copy.
- `escapeHtml` wraps every user-supplied string before interpolation. The test suite injects `<img src=x>` and `<script>` into party names, reasons, and spec files to prove the card never emits raw tags.

No new event types or sink types.

## Acceptance Checks

- `bun test test/telegram-card-builder.test.ts` covers: daemon card rendering, `(all parties)` label for `partyName: "*"`, specFile/runId/error inclusion, HTML injection escape, distinct header + flavor per event type.
- `bun test test/setup-detection.test.ts` covers: env-only detection, OpenClaw config import, malformed OpenClaw JSON tolerated.
- `bun ./scripts/canaries/daemon-events.ts` still passes end-to-end.
- Manual: `quest party {create,bonfire,resume} --name <temp>` with the HTML sink configured lands three cards in Telegram with distinct headers, flavor lines, and a local clock.

## Test Plan

Unit tests above. No new canaries — the existing `daemon-events.ts` canary covers the dispatcher path that the Telegram handler piggybacks on. Manual smoke via a throwaway party lifecycle before every release touching the formatter.

## Rollout / Migration

Zero-config for existing installations: the card formatter only activates on sinks where the operator explicitly sets `parseMode: "HTML"`. Sinks without that field keep the plain-text formatter. No database migration.

## Rollback / Recovery

If the card formatter regresses on a deployed install, flip the sink back to plain text:

```sh
quest observability telegram upsert --id daemon-telegram --chat-id <chat-id> --bot-token-secret-ref <ref>
# omitting --parse-mode clears it, falling back to formatSinkTextMessage
```

If the OpenClaw credential import path misbehaves (e.g., hostile JSON), delete the detected fields manually from `~/.quest-runner/observability/config.json` and re-run `quest observability telegram upsert` with explicit flags. Detection is idempotent; subsequent setup runs re-offer the import.
