# Training Grounds v2

This document describes the intended future version of worker calibration beyond the current built-in `training-grounds-v1` suite.

The current system proves that `quest-runner` can score a worker on real throwaway tasks. v2 should turn that into a configurable training ground that users can extend to match their own workflows.

## Why v2 Exists

The current calibration model is intentionally small:

- one built-in suite
- one fixture shape
- one scoring model
- no user-defined suites
- no external benchmark installation flow

That is enough to validate the architecture, but it is not enough for long-term worker evaluation.

Future users will want to answer questions like:

- Is this worker good at my stack, not just the built-in sample repo?
- Can I benchmark Codex against Hermes or Claude on the same suite?
- Can I reuse public benchmarks like SWE-bench?
- Can I define tests that reflect my actual engineering workflow?
- Can I run periodic recalibration after changing model, prompt, tool policy, or auth mode?

v2 should solve those without creating a second orchestration engine.

## Non-Negotiable Design Rule

Training grounds must keep reusing the normal quest primitives:

- worker registry
- quest specs
- run planning
- run execution
- acceptance checks
- persisted run state

The training ground is a higher-level product surface, not a separate runtime.

If a future implementation starts inventing a duplicate run model or a parallel worker state machine, it is going in the wrong direction.

## Product Goals

1. Let a user pick which tests a worker should take.
2. Let a user install benchmark packs such as SWE-bench or repo-local training packs.
3. Let a user author custom tests based on their own workflow.
4. Produce stable scoring that can update worker trust, XP, and calibration history.
5. Keep every run inspectable and reproducible.

## v2 Scope

### 1. Suite Catalog

Training grounds should support a catalog of suites, where each suite has:

- `id`
- `name`
- `description`
- `kind`
  - `builtin`
  - `installed-pack`
  - `local-custom`
- `disciplines`
- `estimatedCost`
- `estimatedDuration`
- `requirements`
  - toolchain
  - runtime
  - model/backend restrictions

Examples:

- `training-grounds-v2`
- `swe-bench-lite`
- `repo-refactor-pack`
- `internal-qa-regression-pack`

### 2. Installable Benchmark Packs

Some suites should come from installable packs rather than being hardcoded in the repo.

That pack model should support:

- package metadata
- fixture generation
- quest spec templates
- scoring logic
- dependency checks
- optional dataset download/bootstrap steps

Examples of external packs:

- SWE-bench-style tasks
- internal bugfix packs
- framework-specific packs
- repo-maintainer packs

Important constraint:

Installing a pack must not silently run untrusted code inside the main repo. Packs should install into quest-runner-managed state, and any fixture/build steps must be explicit and inspectable.

### 3. User-Authored Tests

Users should be able to create their own test packs without editing quest-runner source.

The minimum user-authored pack should support:

- a manifest file
- one or more fixture templates
- one or more quest specs
- acceptance checks
- scoring weights
- optional per-backend restrictions

Likely format:

- `pack.json` or `pack.yaml`
- `fixtures/`
- `specs/`
- `scoring/`

This should allow a user to encode things like:

- "fix a failing TypeScript test in a Bun repo"
- "refactor a React component without breaking snapshot tests"
- "update docs after a code change"
- "repair CI config drift"

### 4. Training Sessions

The user should be able to create a training session rather than only a single calibration run.

A session should define:

- target worker
- chosen suites
- suite order
- repeat count
- dry-run vs real-backend mode
- whether scoring mutates the worker immediately or only produces a report

This allows:

- probation runs for new workers
- promotion/recalibration runs for existing workers
- A/B comparisons across multiple workers

### 5. Scoring Model

v2 scoring should stay explicit and inspectable.

A calibration result should be derived from persisted facts such as:

- slice completion rate
- acceptance-check pass rate
- retry count
- timeout rate
- diff scope
- conflict rate
- integration success
- token/cost envelope
- elapsed time

Scores should be broken down at least by:

- overall score
- coding
- testing
- docs
- speed
- reliability
- merge safety

The user should be able to configure score weights per suite.

## TUI Requirements

The future TUI should let the user:

1. pick a worker
2. pick one or more suites
3. inspect suite requirements
4. install missing benchmark packs
5. choose whether the result mutates worker trust/XP
6. run the session
7. inspect a scoreboard and drill into failed slices

The TUI should also support:

- "create custom suite"
- "clone existing suite"
- "re-run last training session"
- "compare workers on the same suite"

## Proposed Data Model

### Training Pack

Likely persisted fields:

- `id`
- `version`
- `kind`
- `source`
- `displayName`
- `description`
- `requirements`
- `suites`

### Training Suite

Likely persisted fields:

- `id`
- `packId`
- `displayName`
- `description`
- `fixtureStrategy`
- `questTemplate`
- `scoreWeights`
- `allowedRunners`
- `tags`

### Training Session

Likely persisted fields:

- `id`
- `workerId`
- `suiteIds`
- `status`
- `createdAt`
- `startedAt`
- `completedAt`
- `runIds`
- `result`

### Worker Calibration History

Existing worker calibration history should remain the final source of worker-facing progression state, but session-level records should also be persisted separately so users can inspect training without mutating worker trust.

## SWE-bench And External Benchmarks

SWE-bench support should be treated as a pack integration, not a core special case.

That means:

- install or download happens through a benchmark pack installer
- dataset handling stays outside the core worker schema
- fixture generation produces normal quest-runner source repos/workspaces
- scoring maps benchmark-specific results back into normal calibration records

This matters because SWE-bench is only one benchmark family. The architecture should be general enough to support:

- SWE-bench
- repo-specific bug corpora
- internal regression packs
- benchmark packs published by other teams

## Security And Trust Boundaries

v2 adds real risk because benchmark packs and custom suites may come from outside the current repo.

Required guardrails:

- explicit install step
- inspectable pack metadata
- managed install root under quest-runner state
- no implicit execution during install
- fixture creation isolated from the user’s main checkout
- no secret leakage into pack manifests, prompts, or logs
- signed or checksummed packs if remote distribution is added

## Open Questions

These should be resolved before implementation:

1. Should pack installation be npm-based, file-based, git-based, or all three?
2. Should custom suite authoring be JSON/YAML only, or support code hooks?
3. How much scoring logic can be declarative before users need code?
4. How do we prevent pack-specific logic from turning the core into a plugin host too early?
5. Should sessions update worker trust by default, or require explicit promotion?

## Suggested Implementation Order

1. Add training pack/session schemas.
2. Add suite catalog and `quest workers calibrate --list-suites` expansion.
3. Add pack installation root and local pack loading.
4. Add custom local suite authoring format.
5. Add TUI flow for selecting worker + suites.
6. Add optional trust/XP mutation policy.
7. Add one external benchmark pack such as SWE-bench-lite.

## v2 Success Criteria

v2 is successful if:

- a user can install a benchmark pack without editing quest-runner code
- a user can author a custom suite for their own workflow
- a user can run a multi-suite training session from the TUI
- results are inspectable through normal run/session state
- worker trust updates remain explicit and reversible
- the training ground still reuses the normal quest engine underneath
