# Worker Role Separation v1

## Context

`quest` currently treats one assigned worker as the owner of both implementation and testing by implication. The slice state has one assigned worker, and the execution engine moves that slice into a built-in testing phase after the worker finishes.

That was acceptable for v0, but it is not enough for the operating model we want:
- party-like worker composition
- workers with distinct strengths
- explicit tester identity
- explainable routing for encounters and trials

Earlier prototypes made the need clear: execution, testing, and merge state need distinct ownership, or completion and merge state can drift too easily. The fix is not more flavor. The fix is stronger role boundaries backed by durable state.

## Problem

The current model overloads one worker assignment with too many meanings:
- who builds the slice
- who is best at testing
- who should be credited in summaries
- who should be selected when the slice discipline is not coding

That creates three problems:
- the planner cannot express a dedicated tester
- worker progression and calibration stay too vague
- operator output hides who owned the encounter versus who owned the trial

## Goal

Introduce explicit builder and tester assignment into the plan and run model while keeping integration system-owned.

v1 should make it possible to say:
- worker A builds the encounter
- worker B owns the trial
- the system owns integration

## Non-Goals

This version does not:
- add an integrator worker role
- move final integration into a worker backend
- replace acceptance checks with LLM-only testing
- require a distinct tester when only one hybrid worker exists

## Domain Impact

### Worker roles

Workers gain an explicit role:
- `builder`
- `tester`
- `hybrid`

Meaning:
- `builder` can be assigned to encounters
- `tester` can be assigned to trials
- `hybrid` can do both

### Slice preferences

Slices keep the existing builder preferences:
- `preferredWorkerId`
- `preferredRunner`

Slices gain matching tester preferences:
- `preferredTesterWorkerId`
- `preferredTesterRunner`

### Planned slice state

Planned slices and persisted run slices should carry:
- builder worker id and runner
- tester worker id and runner

The existing assigned worker fields remain the builder alias for compatibility until a later cleanup.

## Workflow

For each slice:
1. Select a builder using the slice discipline and builder-compatible workers.
2. Select a tester using testing and merge-safety signals and tester-compatible workers.
3. Prefer a different tester from the builder when a good option exists.
4. Allow the same hybrid worker to own both phases when no distinct tester is available.
5. Keep integration and turn-in system-owned.

## Planning Rules

Builder scoring should continue to emphasize:
- slice discipline
- trust
- speed
- merge safety
- context endurance
- resource cost

Tester scoring should emphasize:
- testing
- merge safety
- trust
- context endurance
- lower resource cost
- independence bonus when tester is different from builder

Compatibility rules:
- disabled workers are never eligible
- builder assignment requires role `builder` or `hybrid`
- tester assignment requires role `tester` or `hybrid`
- preferred runner constraints apply independently for builder and tester

## Run State

Run state should make phase ownership explicit:
- `assignedWorkerId` and `assignedRunner`
  builder compatibility alias for now
- `assignedTesterWorkerId`
- `assignedTesterRunner`

Operator views should show both:
- builder
- tester

## Execution Model

v1 does not introduce a second testing runtime.

Execution remains:
1. builder backend executes the encounter
2. built-in slice acceptance checks run
3. tester assignment is persisted and surfaced in events and summaries

That means tester identity becomes explicit before we move to a true separate tester backend lane.

## Why This Increment

This is intentionally incremental because Grind showed two important things:
- phase separation is necessary
- merge and lifecycle correctness matter more than theatrical autonomy

Making tester assignment explicit now improves planning and explainability without destabilizing the execution engine.

## Observability

Planner explanation and run summaries should surface:
- chosen builder
- chosen tester
- builder candidates
- tester candidates

Events may continue using the existing testing event names, but testing events should include tester identity when available.

## Acceptance Checks

This feature is correct when:
- workers can be registered with explicit roles
- plans assign both builder and tester for schedulable slices
- run state persists tester assignment
- pretty output and summaries show builder and tester separately
- a hybrid worker still works as a single-worker fallback

## Test Plan

Automated coverage should include:
- builder-only worker excluded from tester assignment
- tester-only worker excluded from builder assignment
- hybrid worker can satisfy both phases
- preferred tester routing
- planner explanation surfaces both candidate lists
- run state persists tester assignment

## Canary Plan

Run at least one disposable real quest where:
- a Codex or OpenClaw builder is selected
- a Hermes or hybrid tester is selected
- the encounter executes normally
- summaries and logs show the tester assignment explicitly

## Rollout

Roll out in three steps:
1. schema and planner
2. run state and summaries
3. dedicated tester runtime in a later version

## Recovery

If role assignment introduces regressions:
- default all workers back to `hybrid`
- preserve builder alias fields
- keep execution on the existing single runtime path
