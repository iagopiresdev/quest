# Party Bonfire v1

## Goal

Add a global party rest state with two operator commands:

- `quest party bonfire [--reason <text>]`
- `quest party resume`

The feature should fit the existing themed presentation layer while staying operationally strict.

## Scope

- Persist party state in `~/.quest/party-state.json`
- Block new `runs execute` dispatch while the party is resting
- Allow already running encounters, trials, and boss-fight work to finish
- Surface the rest state in pretty output and `runs status` / `runs summary` / `runs list` / `runs watch`

## Non-goals

- Not a force-cancel
- Not per-worker or per-backend freezing
- Not a background daemon
- Not an observability sink event stream in v1

## State model

Persist a single global document:

- `status: active | resting`
- optional `reason`
- `updatedAt`
- audit `events`

The state must survive separate CLI invocations. An in-memory flag would be misleading because Quest is command-driven.

## Idempotency and re-commands

The two commands are intentionally asymmetric:

- `quest party bonfire` is **not** idempotent. Re-running it while already resting updates the stored `reason` and appends a new `party_bonfire_lit` event. This lets operators annotate the current rest ("now waiting on the upstream outage") without forcing a resume + re-bonfire cycle.
- `quest party resume` **is** idempotent. Re-running it while already active is a silent no-op ÔÇö no new event, no state mutation.

Rationale: the rest state carries operator intent (the reason) that changes over time; the active state is uniform and carries no such context, so re-emitting `party_resumed` events would just add noise to the audit log.

## Dispatch rule

Only gate new dispatch boundaries:

- block `quest runs execute`

Do not block:

- `quest run`
- `quest runs status`
- `quest runs watch`
- `quest runs land`
- `quest runs cleanup`
- already-running execution phases

This keeps Bonfire aligned with ÔÇťrest at checkpointÔÇŁ rather than ÔÇťkill everything now.ÔÇŁ

## CLI surface

- `quest party status`
- `quest party bonfire [--reason <text>]`
- `quest party resume`

Pretty mode should render:

- `The party rests at a bonfire.`
- `The party presses on.`

## Validation

- unit test for persisted bonfire/resume transitions
- pipeline test proving `runs execute` fails with `quest_party_resting`
- CLI test for `party bonfire` / `party resume`
- CLI test showing a planned run stays `planned` while the party is resting
