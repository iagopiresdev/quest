# Linear State Tracker v1

## Context

Before this spec, the Linear sink was comment-only. Every observability event for every spec posted one new comment on a single pinned Linear issue configured via `--issue-id`. Useful as a chat-log-in-Linear, useless as a project tracker.

Symphony (OpenClaw's Elixir orchestrator) treats each Linear issue as a work item and moves it through workflow states as the runtime makes progress. Quest Runner now does the same, on top of the existing comment stream. The comment path is unchanged (operators who only want observability comments get them); the state-transition path is a per-spec opt-in.

## Problem

- A Quest-Linear operator wants their team's Linear board to reflect Quest work status automatically: a "deploy-api" issue moves to *In Progress* when the daemon dispatches its spec, to *Done* when it lands, back to *Todo* if it fails.
- The prior sink had no way to know *which* issue corresponds to *which* spec. One sink â†’ one issue â†’ all events.
- Without per-spec issue mapping, `issueUpdate` would move the observability issue itself every time, which is wrong.

## Goal

1. Let specs declare an opt-in `tracker.linear.issueId` block.
2. Thread that id through `ObservableDaemonEvent.trackerIssueId` so sinks can act on it.
3. Teach the Linear sink to issue `stateLookup` + `issueUpdate` GraphQL mutations on `daemon_dispatched` / `daemon_landed` / `daemon_failed`, alongside the existing `commentCreate`.
4. Per-sink `stateMap` config to override the default *In Progress* / *Done* / *Todo* names when teams use different workflow vocabularies.
5. Comment-only behavior stays the default for specs without a tracker block.

## Non-Goals

- No per-slice state transitions. Only whole-spec lifecycle.
- No issue creation per run. Quest does not spawn new Linear issues.
- No label / assignee / priority updates.
- No Jira / GitHub-Issues tracker in v1. The `tracker` block is structured so they can slot in later.
- No bidirectional sync (Quest does not read state back from Linear).

## Domain Impact

### Spec schema â€” new `tracker` block

`src/core/planning/spec-schema.ts`:

```ts
export const questTrackerSchema = z
  .object({
    linear: z
      .object({
        issueId: nonEmptyString(120),
      })
      .strict()
      .optional(),
  })
  .strict();

// questSpecSchema gets:
tracker: questTrackerSchema.optional();
```

Strict mode means unknown keys (e.g. `tracker.jira`) reject â€” we can add them later without a breaking change because the top-level field is already optional.

### Observability event â€” new `trackerIssueId` field

`src/core/observability/observable-events.ts`:

```ts
// ObservableDaemonEvent:
trackerIssueId: nonEmptyString(120).nullable().default(null);
```

`createObservableDaemonEvent({ ..., trackerIssueId })` coerces empty strings to null so legacy callers keep working unchanged.

### Daemon tick â€” thread it through

`src/core/daemon/tick.ts` extracts `candidate.document.tracker?.linear?.issueId ?? null` once per spec and passes it to the three lifecycle event emissions (`daemon_dispatched`, `daemon_landed`, `daemon_failed`).

Party-admin and budget events (`daemon_party_created` / `resting` / `resumed` / `daemon_budget_exhausted`) do NOT carry a tracker id â€” they're not per-spec transitions.

### Linear sink â€” new state-transition path

`src/core/observability/sinks/linear-sink.ts`:

- New schema fields:
  - `stateMap?: { dispatched?, landed?, failed?, testing?, in_review?, blocked? }` (string | null each)
- On delivery:
  1. Post the comment (existing behavior, unchanged).
  2. If the event carries a `trackerIssueId` AND its eventType maps to a lifecycle state AND the resolved `stateMap` entry is non-null: run `stateLookup(issueId, stateName)` â†’ `issueUpdate(issueId, stateId)`.
  3. State lookup failure or issue-update failure returns a `failed` delivery with `"tracker state transition: <reason>"` so the delivery queue can retry.
- Event type â†’ state key mapping:
  - `daemon_dispatched` â†’ `dispatched` â†’ default *In Progress*
  - `run_integration_checks_started` â†’ `testing` â†’ default *Testing*
  - `run_integration_checks_completed` â†’ `in_review` â†’ default *In Review*
  - `run_integration_checks_failed` â†’ `blocked` â†’ default *Blocked*
  - `run_blocked` â†’ `blocked` â†’ default *Blocked*
  - `daemon_landed` â†’ `landed` â†’ default *Done*
  - `daemon_failed` â†’ `failed` â†’ default *Todo*
- `null` entries in `stateMap` opt out: `stateMap: { failed: null }` leaves the issue alone on daemon failure but still moves it on dispatch/testing/review/landing.

### CLI â€” new flags

```
--state-dispatched <name>  # default "In Progress", pass "none" to opt out
--state-testing <name>     # default "Testing"
--state-in-review <name>   # default "In Review"
--state-blocked <name>     # default "Blocked"
--state-landed <name>      # default "Done"
--state-failed <name>      # default "Todo"
```

## User Or Operator Workflow

### Spec-level opt-in (per issue)

Pin a spec to a Linear issue:

```yaml
version: 1
title: Ship checkout flow
workspace: checkout
tracker:
  linear:
    issueId: ENG-142
slices:
  - id: build
    title: Build checkout
    ...
```

Configure the sink once:

```sh
quest observability linear upsert \
  --id project-tracker \
  --issue-id ENG-OBSERVABILITY \
  --api-key-secret-ref quest-linear-token \
  --events daemon_dispatched,daemon_landed,daemon_failed \
  --state-landed "Ready for Review" \
  --state-failed "Blocked"
```

As the spec progresses, `ENG-142` walks through:
1. `daemon_dispatched` â†’ *In Progress* (comment + state move)
2. `run_integration_checks_started` â†’ *Testing* (comment surfaces `checkCount`)
3. `run_integration_checks_completed` â†’ *In Review* (comment surfaces `checkCount`)
4. `daemon_landed` â†’ *Done*

If tests fail: `run_integration_checks_failed` â†’ *Blocked*, comment surfaces the failing `command` + `exitCode`. If the run is blocked externally: `run_blocked` â†’ *Blocked*.

### Party-level fallback (per inbox)

For workflows where every spec in a party maps to the same Linear issue (shared-issue execution), declare the default once at party creation:

```sh
quest party create --name alpha \
  --source-repo ~/code/myrepo \
  --target-ref main \
  --linear-default-issue-id FRI-142
```

Specs dropped into alpha's inbox without their own `tracker.linear.issueId` will inherit `FRI-142`. Specs that declare their own issue id still override â€” spec-level always wins over party-level fallback. The fallback is stamped onto the running document at dispatch time so run-level events (testing, in_review, blocked) inherit it automatically.

## Data And State Changes

- `ObservableDaemonEvent` gains `trackerIssueId: nullable`. Existing stored delivery records are backward compatible (new field defaults to null).
- `LinearSink` gains `stateMap: optional`. Existing sinks without it keep behaving exactly as before (comment-only; no state transitions because default behavior requires a `trackerIssueId` on the event).
- No migration required.

## Command Surface

No new top-level commands. Three new flags on `observability linear upsert` (listed above).

## Observability

The sink itself is the observability surface; the spec ships tests that hit a mock Linear GraphQL server asserting:
- Default state map resolves the right state name per event type.
- `stateMap` overrides win over defaults.
- `null` entries in `stateMap` opt out cleanly.
- Events without `trackerIssueId` skip transitions entirely.
- Party-admin events never trigger transitions even if a tracker id somehow leaked through.
- State lookup returning no nodes fails the delivery with a specific `state_not_found:<name>` error.

## Rollout / Migration

Zero-config for existing installs. Specs without `tracker.linear` stay comment-only. Existing Linear sinks without `stateMap` use default transitions only when a tracker id appears in an event â€” which requires the spec opt-in, which requires the operator to deliberately add it. No silent behavior changes.

## Rollback / Recovery

Remove the `tracker` block from the spec to stop moving the Linear issue. No code flip needed.

If the Linear workflow has a state named differently than the defaults (common â€” e.g. "To Do" vs "Todo"), configure `--state-failed "To Do"` etc. State lookup is name-exact via the Linear GraphQL `filter: { name: { eq } }` predicate.
